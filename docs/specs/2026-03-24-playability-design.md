# 可玩性增强设计

## 问题

系统都搭好了，但没有正在发生的事需要玩家参与。缺少"每天想打开看看"的动力。

## 设计目标

- 打开先看摘要再行动
- 事情密度按离线时长自动调整
- 选择有后果、有新鲜事、有事要处理

## 一、开屏简报

打开游戏不直接进场景，先弹出"今日简报"面板：

- 标题：距上次来过了多久（"你离开了 3 小时" / "你离开了 2 天"）
- 发生的事：追算期间触发的事件列表
- 待处理：需要玩家做决定的事项
- 故事进展：有推进的故事线摘要
- 底部按钮："好的，开始" → 关闭简报进入场景

数据来源：现有 catch-up 的 summary + 待办列表 + 故事进展。

## 二、待办事项系统

从事件和世界状态中生成具体的待处理任务。

### 待办类型

**老师线：**

- 批改作业（每天下午自动产生，不处理会积压）
- 备课（每周一早上产生）
- 处理学生问题（学生打架、逃学等事件触发）
- 接待家长（家长来访事件触发）
- 写期末评语（学期末触发）

**邮递员线：**

- 送信/送报（每天早上产生待投递邮件列表）
- 处理问题邮件（地址错误、破损等事件触发）

**通用：**

- 回复某人的请求（NPC 对话中产生）

### 数据模型

新表 `todos`：

- id, type, title, description, npcId?, priority, status(pending/done/expired), createdDate, deadlineDate

### 生成规则

在 tick 中按条件生成待办：

- 离线 1-3 小时：1-2 个待办
- 离线 3-12 小时：2-4 个
- 离线 12-48 小时：4-6 个
- 离线 48 小时以上：上限 8 个
- 过期未处理的待办会产生后果

### 处理方式

玩家走到对应地点 + 点击相关 NPC 或物品即可处理。比如：

- "批改作业" → 走到办公室 → 点击桌子 → 自动完成 + AI 生成一段描述
- "处理学生打架" → 走到操场/教室 → 点击涉及的学生 → 对话 + 选择

## 三、事件密度调整

当前 tick 中每步都可能产生事件，但没有控制密度。改为：

- 追算时计算应产生的事件数 = f(离线时长)
- 在追算步骤中均匀分布事件触发点
- 不是每个 tick 都产生事件，而是按目标数量分配

## 四、选择后果系统

让玩家的选择产生可见的后续影响。

### 后果触发器

每个选择（encourage/criticize/ignore）在 `player_choices` 中记录。后果系统在后续 tick 中检查最近的选择，按规则触发后果事件：

- 鼓励学生 → 3-7 天后触发"该学生成绩进步"或"该学生更积极"事件
- 批评学生 → 3-7 天后触发"该学生变沉默"或"该学生改正了"事件
- 忽略 → 无直接后果，但亲密度不变（可能错过机会）

### 后果事件

后果事件是特殊类型的事件，包含：

- 触发条件：距离选择 N 天后
- 关联 NPC
- 正面/负面描述（AI 生成）
- 对 NPC 状态的影响（亲密度、心情）

后果在简报中突出显示："因为你上次鼓励了铁柱，他这周的作业写得认真多了。"

## 技术方案

### 新增文件

- `server/db/schema.ts` — 添加 todos 表
- `server/db/queries.ts` — todos CRUD
- `server/engine/todos.ts` — 待办生成逻辑
- `server/engine/consequences.ts` — 选择后果逻辑
- `src/components/Briefing.vue` — 开屏简报组件
- `src/api/client.ts` — 添加 todos API

### 新增 API

- GET /api/briefing — 返回简报数据（事件摘要 + 待办 + 故事进展）
- POST /api/todo/complete — 标记待办完成
- GET /api/todos — 获取待办列表

### 修改文件

- `server/engine/tick.ts` — 集成待办生成和后果检查
- `server/engine/catch-up.ts` — 事件密度控制
- `src/App.vue` — 加入简报流程
- `src/components/SidePanel.vue` — 显示待办列表
